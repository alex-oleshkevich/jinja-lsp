## 1. Purpose & Scope

This spec defines how we test complete journeys through the **real** `jinja-lsp` binary — both the CLI and the LSP server, exercised the way a user's editor and CI would. It is the authority every feature's **End-to-End Test Plan** (§12) defers to. The two branches share one principle: drive the real binary, never a stub.

This spec covers:

- The two e2e branches — Branch A (golden-file `check`) and Branch B (`pytest-lsp` protocol journeys).
- The tools and harness for each.
- The coverage policy every user-facing feature meets.
- The shared protocol-conformance journeys.

Out of scope: unit and integration testing, and the fixtures registry — those live in [E17-testing](E17-testing.md).

## 2. Coverage policy

Every user-facing feature's e2e plan covers its full visible scope, on both branches where they apply.

**REQ-E2E-01 — Cover 100% of feature scope, end to end.**

Each user-facing feature's §12 covers all of its user-visible scope: the happy path **and** every reasonably possible error path — undefined symbols, missing templates, invalid config, empty workspaces. A journey a real user (or CI) can hit must have a scenario with an asserted outcome.

**REQ-E2E-02 — Both branches are first-class.**

Diagnostics are verified by **Branch A** (golden `check` files) *and* must match what the server publishes via **Branch B**. An e2e plan that exercises only one branch where both apply is incomplete. CLI/server parity ([E01](E01-architecture.md) REQ-ARCH-01) is asserted, not assumed.

## 3. Branch A — golden-file `check`

Branch A is the diagnostics regression gate. A Rust test runs the real CLI over each fixture and diffs the machine-readable output against the fixture's golden file.

**REQ-E2E-03 — `check --format json` is diffed against the golden file.**

A `cargo test` (run under `cargo nextest`) invokes `jinja-lsp check --format json` against each fixture directory in the registry ([E17](E17-testing.md#5-fixtures-registry)) and diffs the actual output against that fixture's `expected-diagnostics.json`. On mismatch, the failure shows a unified diff of expected vs actual. Because the JSON shape is identical across the CLI, the server, and the golden file ([E17](E17-testing.md#4-the-golden-file-shape)), a single golden file gates both front-ends.

**REQ-E2E-04 — Golden files update behind an env flag.**

Golden files are regenerated by re-running with an update flag — `UPDATE_FIXTURES=1 cargo nextest run`. Without the flag, a mismatch fails the test. This keeps golden updates deliberate: a developer sees the diff, decides it's correct, then re-records.

## 4. Branch B — `pytest-lsp` protocol journeys

Branch B drives the real LSP server over stdio, exercising the interactive features the way an editor does. We do not hand-roll a client.

**REQ-E2E-05 — Drive the real `jinja-lsp lsp` binary with `pytest-lsp`.**

The Python suite in `tests/e2e/` uses [`pytest-lsp`](https://github.com/swyddfa/lsp-devtools) with `lsprotocol` to drive the real `jinja-lsp lsp` stdio binary ([E03](E03-tech-stack.md)). `pytest-lsp` handles capability negotiation and JSON-RPC plumbing — a mature, server-language-agnostic client fixture. Hand-rolling an LSP client harness is explicitly forbidden. This is the **only** Python in the repository, confined to `tests/e2e/`.

**REQ-E2E-06 — Branch B test cases.**

Branch B covers, at minimum, these protocol journeys against the `starlette-blog` fixture:

| Journey | Assertion |
|---|---|
| `didOpen` → `publishDiagnostics` | published findings match the golden file (parity with Branch A) |
| `completion` + `completionItem/resolve` | expected items returned; full doc attached on resolve ([F05](../features/F05-completions.md)) |
| `hover` on a built-in filter | doc substring present ([F06](../features/F06-hover.md)) |
| `signatureHelp` inside a macro call | active parameter correct ([F07](../features/F07-signature-help.md)) |
| `definition` on a macro call | `Location` points at the macro definition ([F08](../features/F08-go-to-definition.md)) |
| `codeAction` at a diagnostic | the expected fix is offered ([F17](../features/F17-code-actions.md)) |

## 5. Environment, seeding & teardown

Both branches seed state from the fixtures registry and start each scenario clean.

- **Test data** — every scenario seeds from a fixture workspace in [E17](E17-testing.md#5-fixtures-registry); no scenario invents its own templates inline.
- **Isolation** — each Branch B scenario starts a fresh server process over a copy of the fixture; each Branch A run is a fresh CLI invocation.
- **Teardown** — the server process is shut down per scenario; no state leaks between journeys.

## 6. Shared protocol-conformance journeys

Some journeys belong to no single feature — they test the protocol contract itself. They live here and run on Branch B.

**REQ-E2E-07 — Protocol-conformance journeys.**

These shared journeys exercise the [E01](E01-architecture.md) lifecycle and protocol conduct: capability negotiation on `initialize`, an empty publish on a clean open, an empty publish when the last finding in a file is fixed, coalesced relinks under rapid `didChange`, and config-reload via `didChangeWatchedFiles` ([E15](E15-app-config.md)). They are owned here, not by any feature, because they assert the contract every feature wires into.

## 7. Conventions

- **Naming** — scenarios are `E2E-NN` within a feature, titled by the journey.
- **Structure** — given (seeded fixture) → when (CLI run or LSP request) → then (asserted golden/response).
- **Waiting** — Branch B waits on the awaited notification/response, never a fixed sleep.
- **Where feature e2e plans link** — every feature's §12 links to §2 here for the coverage policy rather than restating it.

## 8. Running e2e & CI

Branch A runs as part of `cargo nextest run` — no separate step. Branch B runs under `pytest` in `tests/e2e/`, in a dedicated CI job that sets up Python and installs `pytest-lsp` ([F21](../features/F21-release-ci.md)). Both gate merge: a golden mismatch or a failed protocol journey blocks the PR. Branch B captures the JSON-RPC trace on failure for debugging.
